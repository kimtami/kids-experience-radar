# 고양시 뉴스 일일 증분 수집 설계

검토일: 2026-07-15 (KST)

## 관측 결과와 보장 범위

고양시 뉴스 검색은 현재 99개 기사·10페이지다. 목록과 상세 응답 모두 `ETag`,
`Last-Modified`, `Cache-Control`을 제공하지 않았다. 따라서 기사 본문의 표시 없는 수정까지
매일 즉시 감지하려면 상세 99건 전수 조회가 필요하다. 현재 구현은 이 무손실 모드를
사용하며 기본 5초 호스트 간격에서 9분 8초가 걸렸다.

운영 시간이 중요해지면 다음 증분 모드를 추가한다. 이 모드는 목록 누락은 매일 탐지하고,
종료·무관 기사 본문의 표시 없는 수정은 최대 7일 이내 탐지하는 명시적 절충이다.

## 증분 알고리즘

1. 목록 10페이지를 매일 전수 조회하고 total, page count, 행 수, 중복 ID를 검증한다.
2. 신규 게시물, 목록 지문 변경, 파서 버전 변경, 손상 캐시, 재검증 기한 도래 건만
   상세 GET을 수행한다.
3. 진행 중·예정 프로그램을 포함한 게시물은 매일 재검증한다.
4. 무관·종료 게시물은 `post_id` 해시로 요일을 분산해 7일 안에 재검증한다.
5. 재검증 대상 상세 요청이 실패하면 오래된 캐시로 성공 처리하지 않고 소스 전체를
   fail-loud 처리한다.
6. 캐시 적중 상세도 매일 실제 상세 URL에 대한 robots 판정을 다시 수행한다.
7. HTML은 저장하지 않고 허용된 `GoyangProgramFact` 사실 필드 또는 빈 결과만 저장한다.
8. 전체 목록·필수 상세가 모두 성공한 뒤 하나의 트랜잭션으로 캐시를 교체한다.

예상 정상 요청량은 `목록 10 + 진행/예정 P + (99-P)/7 + 신규·변경 N`이며 보통
24~30회, 약 2~2.5분이다. 최초 실행과 파서 버전 변경 직후에는 의도적으로 전수
재검증한다.

## 구현 경계

- `store.py`: `source_item_cache(source_id, item_id, list_fingerprint,
  parser_version, payload_json, fetched_at, next_refresh_at)`와 원자적 load/replace API
- `sources/base.py`: 기존 소스 인터페이스를 건드리지 않는 작은 `CacheAwareSource` seam
- `engine.py`: cache-aware 소스만 `crawl_cached(client, window, store)`로 실행
- `goyang_children_museum.py`: 목록 전수 검증과 상세/캐시 처리를 분리하고, 캐시 사실도
  현재 `_now()`를 기준으로 다시 Event에 매핑

캐시 재생 이벤트도 매일 기존 `upsert_events()`를 거쳐 `last_seen`을 갱신한다. 목록
완전성 오류, 필수 재검증 실패, 잘못된 source ID, 중복 item ID, 손상 JSON에서는 기존
캐시를 보존하고 실행을 실패시킨다.

## 필수 회귀 테스트

- 차가운 캐시 전수 상세, 따뜻한 캐시 목록 전수·상세 생략
- 신규/지문 변경/파서 버전 변경/손상 캐시의 선택적 재조회
- 진행·예정 매일 재조회와 종료·무관 7일 분산 재조회
- stale fallback 금지, 목록 불완전 시 캐시 무변경, 성공 후 원자 교체
- 캐시 재생의 `last_seen` 갱신과 현재시각 상태 재계산
- HTML·이미지·첨부·휴대폰 미저장
- 캐시 적중 상세 URL의 현재 robots 거부 시 fail-loud
